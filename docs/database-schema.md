# AWS DynamoDB Single-Table Database Schema Design

This document describes the DynamoDB Single-Table schema design for Kesahomma26 (Issue #13). The design enforces **strict tenant isolation** at the database level by partition key design, models all entities in a single table, and optimizes costs by avoiding Global Secondary Indexes (GSIs) for the MVP.

## Design Principles

1. **Generic Key Names**: The table uses generic names for primary keys:
   - **`PK` (Partition Key)**: String
   - **`SK` (Sort Key)**: String
2. **Explicit Tenant Isolation**: To prevent cross-tenant data access, **every** item in the table uses the User ID (Cognito `sub`) as the prefix or value of the Partition Key (`PK`). 
   - `PK` format: `USER#<userId>`
3. **Hierarchical Sorting**: Sort Keys (`SK`) are structured hierarchically using `#` delimiters. This allows efficient range queries and prefix filtering (e.g., fetching all holdings under a specific portfolio using `begins_with`).

## Entity Mapping Summary

| Entity Type | Partition Key (`PK`) | Sort Key (`SK`) | Key Attributes |
| :--- | :--- | :--- | :--- |
| **User** | `USER#<userId>` | `METADATA` | `email`, `role`, `createdAt`, `updatedAt` |
| **Portfolio** | `USER#<userId>` | `PORTFOLIO#<portfolioId>` | `name`, `description`, `createdAt`, `updatedAt` |
| **Holding** | `USER#<userId>` | `PORTFOLIO#<portfolioId>#HOLDING#<symbol>` | `symbol`, `quantity`, `averagePrice`, `lastUpdated` |
| **Transaction** | `USER#<userId>` | `PORTFOLIO#<portfolioId>#TRANSACTION#<timestamp>#<transactionId>` | `symbol`, `type` (BUY/SELL), `quantity`, `price`, `timestamp` |
| **Analysis Job** | `USER#<userId>` | `PORTFOLIO#<portfolioId>#ANALYSIS_JOB#<jobId>` | `status` (PENDING/RUNNING/COMPLETED/FAILED), `createdAt`, `completedAt` |
| **Analysis Result** | `USER#<userId>` | `PORTFOLIO#<portfolioId>#ANALYSIS_RESULT#<jobId>` | `metrics` (JSON object), `recommendations` (list), `createdAt` |

## Detailed Entity Schemas

### 1. User (`METADATA`)
Stores user-specific settings, account status, and roles.
* **PK**: `USER#<userId>`
* **SK**: `METADATA`
* **Attributes**:
  ```json
  {
    "email": "user@example.com",
    "role": "user",
    "createdAt": "2026-06-10T12:00:00Z",
    "updatedAt": "2026-06-10T12:00:00Z"
  }
  ```

### 2. Portfolio (`PORTFOLIO#<portfolioId>`)
Represents a user's investment portfolio containing multiple holdings.
* **PK**: `USER#<userId>`
* **SK**: `PORTFOLIO#<portfolioId>`
* **Attributes**:
  ```json
  {
    "name": "Tech Growth Portfolio",
    "description": "Long term tech stock investments",
    "createdAt": "2026-06-10T12:05:00Z",
    "updatedAt": "2026-06-10T12:05:00Z"
  }
  ```

### 3. Holding (`PORTFOLIO#<portfolioId>#HOLDING#<symbol>`)
Represents the current consolidated shares/units of a specific stock or asset.
* **PK**: `USER#<userId>`
* **SK**: `PORTFOLIO#<portfolioId>#HOLDING#<symbol>`
* **Attributes**:
  ```json
  {
    "symbol": "AAPL",
    "quantity": 15.5,
    "averagePrice": 172.45,
    "lastUpdated": "2026-06-10T12:10:00Z"
  }
  ```

### 4. Transaction (`PORTFOLIO#<portfolioId>#TRANSACTION#<timestamp>#<transactionId>`)
Historical record of buy/sell actions. The timestamp and unique transaction ID are included in the SK to support sorted history and uniqueness.
* **PK**: `USER#<userId>`
* **SK**: `PORTFOLIO#<portfolioId>#TRANSACTION#<timestamp>#<transactionId>`
* **Attributes**:
  ```json
  {
    "symbol": "AAPL",
    "type": "BUY",
    "quantity": 10,
    "price": 170.50,
    "timestamp": "2026-06-10T12:10:00Z"
  }
  ```

### 5. Analysis Job (`PORTFOLIO#<portfolioId>#ANALYSIS_JOB#<jobId>`)
Tracks the status of an asynchronous portfolio analysis job (e.g. Bedrock analysis queue).
* **PK**: `USER#<userId>`
* **SK**: `PORTFOLIO#<portfolioId>#ANALYSIS_JOB#<jobId>`
* **Attributes**:
  ```json
  {
    "status": "RUNNING",
    "createdAt": "2026-06-10T12:15:00Z",
    "completedAt": null,
    "error": null
  }
  ```

### 6. Analysis Result (`PORTFOLIO#<portfolioId>#ANALYSIS_RESULT#<jobId>`)
Stores the output generated by a completed analysis job.
* **PK**: `USER#<userId>`
* **SK**: `PORTFOLIO#<portfolioId>#ANALYSIS_RESULT#<jobId>`
* **Attributes**:
  ```json
  {
    "metrics": {
      "diversificationScore": 85,
      "riskLevel": "Medium"
    },
    "recommendations": [
      "Consider adding exposure to energy sector.",
      "Reduce concentration in technology stocks."
    ],
    "createdAt": "2026-06-10T12:16:30Z"
  }
  ```

## Access Patterns (AP)

Here are the specific query configurations mapped to API requests.

### AP1: Fetch User Profile
* **Operation**: `GetItem`
* **Key**:
  - `PK = USER#<userId>`
  - `SK = METADATA`

### AP2: Fetch All Portfolios for a User
* **Operation**: `Query`
* **Key Condition**:
  - `PK = USER#<userId>`
  - `SK begins_with PORTFOLIO#`
* *Note: Since holdings and transactions have longer SK paths, this begins_with will match only portfolio metadata rows because they do not have trailing dividers like `#HOLDING#`.*

### AP3: Fetch a Single Portfolio Metadata
* **Operation**: `GetItem`
* **Key**:
  - `PK = USER#<userId>`
  - `SK = PORTFOLIO#<portfolioId>`

### AP4: Fetch All Holdings in a Portfolio
* **Operation**: `Query`
* **Key Condition**:
  - `PK = USER#<userId>`
  - `SK begins_with PORTFOLIO#<portfolioId>#HOLDING#`

### AP5: Fetch a Specific Holding in a Portfolio
* **Operation**: `GetItem`
* **Key**:
  - `PK = USER#<userId>`
  - `SK = PORTFOLIO#<portfolioId>#HOLDING#<symbol>`

### AP6: Fetch All Transactions in a Portfolio (Sorted by Timestamp)
* **Operation**: `Query`
* **Key Condition**:
  - `PK = USER#<userId>`
  - `SK begins_with PORTFOLIO#<portfolioId>#TRANSACTION#`
* *Note: Results are naturally sorted chronologically by DynamoDB because the timestamp is the first element after the TRANSACTION prefix in the Sort Key.*

### AP7: Fetch All Analysis Jobs for a Portfolio
* **Operation**: `Query`
* **Key Condition**:
  - `PK = USER#<userId>`
  - `SK begins_with PORTFOLIO#<portfolioId>#ANALYSIS_JOB#`

### AP8: Fetch Analysis Result for a Specific Job
* **Operation**: `GetItem`
* **Key**:
  - `PK = USER#<userId>`
  - `SK = PORTFOLIO#<portfolioId>#ANALYSIS_RESULT#<jobId>`
